2. **prepare.yml** — Rocky 9 nsswitch fix + testuser creation on ALL platforms:
```yaml
---
- name: Prepare
  hosts: all
  gather_facts: false
  become: true

  tasks:
    - name: Fix Rocky 9 nsswitch for sudo/PAM in containers
      ansible.builtin.command:
        cmd: sed -i 's/ sss//g' /etc/nsswitch.conf
      changed_when: true
      failed_when: false

    - name: Create testuser for audit verification
      ansible.builtin.command:
        cmd: useradd -m testuser
      changed_when: true
      failed_when: false

    - name: Set testuser password expiry to never (chage -M -1)
      ansible.builtin.command:
        cmd: chage -M -1 testuser
      changed_when: true
```

The testuser has UID >= 1000 and `chage -M -1` sets maxdays to -1 (never expires). This means:
- review.yml will identify testuser in _ua_no_expiry
- enforce.yml two-step chage will detect maxdays -1 and set it to 90
- verify.yml can assert that testuser no longer has "never" expiry

3. **converge.yml** — run role in enforce mode:

- name: Converge

4. **verify.yml** — assertions mapped to requirements. All assertions use `ansible.builtin.assert` with `fail_msg` (STD-11):
```yaml
---
- name: Verify
  hosts: all
  become: true
  gather_facts: true

  tasks:
    - name: Verify testuser password expiry is set (USR-03)
      ansible.builtin.command:
        cmd: chage -l testuser
      register: _ua_verify_chage
      changed_when: false

    - name: Assert testuser has password expiry set
      ansible.builtin.assert:
        that:
          - _ua_verify_chage.stdout is search('Maximum number of days between password change\s*:\s*90')
        fail_msg: >-
          testuser maxdays is not 90 after enforce.
          Got: {{ _ua_verify_chage.stdout }}
        quiet: true

    - name: Verify root account not touched (USR-04)
      ansible.builtin.command:
        cmd: chage -l root
      register: _ua_verify_root
      changed_when: false

    - name: Assert root password expiry was not changed to enforcement value
      ansible.builtin.assert:
        that:
          - _ua_verify_root.stdout is not search('Maximum number of days between password change\s*:\s*90')
        fail_msg: >-
          root account maxdays was changed to 90 by enforce tasks.
          This violates USR-04. Got: {{ _ua_verify_root.stdout }}
        quiet: true

    - name: Check testuser lock state via passwd -S (USR-02)
      ansible.builtin.command:
        cmd: passwd -S testuser
      register: _ua_verify_lock
      changed_when: false
      failed_when: false

    - name: Gather package facts for shadow-utils check (STD-10)
      ansible.builtin.package_facts:
        manager: auto

    - name: Assert shadow-utils present (RedHat)
      ansible.builtin.assert:
        that:
          - "'shadow-utils' in ansible_facts.packages"
        fail_msg: shadow-utils package not found on RedHat
        quiet: true
      when: ansible_os_family == 'RedHat'

    - name: Assert login package present (Debian)
      ansible.builtin.assert:
        that:
          - "'login' in ansible_facts.packages"
        fail_msg: login package not found on Debian
        quiet: true
      when: ansible_os_family == 'Debian'
```

Note on USR-02 (lock state): The testuser may or may not be in the inactive list from `lastlog --before 90` depending on whether the container sets a last login record for testuser. The `passwd -S` check is informational — the primary verification is USR-03 (password expiry set) and USR-04 (root untouched). If testuser has never logged in, lastlog --before 90 will include it and the lock will apply; if it has a recent lastlog entry from useradd, it won't be in the inactive list.

Note on USR-05 (idempotence): Molecule's built-in idempotence test (`molecule test` runs converge twice and checks for 0 changes) validates this requirement automatically.
