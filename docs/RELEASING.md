# Releasing & Verifying

The collection ships with a fully signed, verifiable supply chain: every build
is accompanied by a **SLSA provenance attestation** and a **keyless cosign
signature** (Sigstore — Fulcio certificate + Rekor transparency log), and
release tags are **GPG-signed**.

## Cutting a release

1. Promote `[Unreleased]` in `CHANGELOG.md` (or run `antsibull-changelog release`)
   and bump `version:` in `galaxy.yml` (semver: new role → MINOR, fix → PATCH,
   breaking → MAJOR).
2. Commit, then create a **signed, annotated** tag:

   ```bash
   git tag -s v1.3.0 -m "malpanez.security 1.3.0"
   git push origin v1.3.0   # only with explicit approval
   ```

   Configure tag signing once: `git config tag.gpgSign true` and
   `git config user.signingkey <key-id>`.
3. CI builds the collection, attaches the SLSA provenance attestation, and signs
   the tarball with cosign (keyless). Artifacts: `*.tar.gz`, `*.tar.gz.sig`,
   `*.tar.gz.pem`.

## Verifying a downloaded artifact

**Provenance (who/where/how it was built):**

```bash
gh attestation verify malpanez-security-1.3.0.tar.gz --repo malpanez/security
```

**cosign signature (keyless / Sigstore):**

```bash
cosign verify-blob \
  --certificate        malpanez-security-1.3.0.tar.gz.pem \
  --signature          malpanez-security-1.3.0.tar.gz.sig \
  --certificate-identity-regexp 'https://github.com/malpanez/security/.+' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  malpanez-security-1.3.0.tar.gz
```

**Tag signature:**

```bash
git verify-tag v1.3.0
```

## SBOM

An SPDX SBOM is generated by the Security Scanning workflow (`security-scan.yml`,
`Generate SBOM`) and uploaded as a build artifact for each run.
